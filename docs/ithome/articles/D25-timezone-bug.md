# Day 25｜AI 時代怎麼修一個 bug：從 -8 小時的時區偏移說起

> 這是系列第 25 天。昨天講了四個小 change 的小步快跑，今天用一個真實 bug 走完整趟「AI 協作下的除錯流程」：**AI 負責病理解剖與列選項，人負責決策，SDD 負責不讓「順手修一修」變成災難。**

## 案發現場

準備開工「每週重複時段」功能時，先撞到一件怪事：**教練建立 10:00 的時段，畫面顯示 02:00。** 固定差 8 小時，台灣是 UTC+8——這個數字太可疑了。

未來預計建立一個新功能，讓教練可以批次新增固定開課的時間，這時候時區問題必須先處理好，今天就以這個 bug 為例來探討如何與 AI 一起追本溯源！

## 第一個決策：不順手修

我做的第一件事不是打開 code，而是問一個流程問題：**這個 bug 跟新功能是什麼關係？**

答案是強耦合——「每週二 10:00 重複開課」建立在 wall-time 語意上，時區地基不修，上面蓋什麼都是歪的。所以它不是「順手修」，而是被正式納入這個 change 的**前置任務群**：proposal 裡寫明「時區修正（前置）」，tasks.md 把它排在第一組，驗收通過才准動新功能。

這就是 SDD 在除錯裡的角色：它不幫你修 bug，它防止你**在錯的地基上繼續蓋樓，然後分不清是地基歪還是新樓歪**。

## explore：讓 AI 做病理解剖

接著把症狀丟給 AI：「建 10:00 顯示 02:00，固定偏 -8h」。幾輪追查後，病理報告出來——三個**各自都沒錯**的環節，串成一個錯誤：

1. **欄位是 `TIMESTAMP`（無時區）**——Prisma `DateTime` 在 PostgreSQL 的預設對應
2. **App 寫入 `toISOString()`**——值是 UTC，標準做法
3. **PostgREST 讀回無時區欄位時不帶 `Z` 後綴**——而 JS 的 `new Date("2026-07-07T02:00:00")` 把無時區標記的字串**當本地時間解析**

寫入語意是 UTC、讀回語意變本地 → 在 UTC+8 固定偏 -8h。

這段是 AI 在除錯裡真正的強項：**跨層知識的即時串接**。Prisma 的型別對應、PostgREST 的序列化行為、ECMAScript 的日期解析規則——三個分屬不同文件的冷知識，人類去查要翻半天，AI 一次攤開。但是我們在每一個功能開發時都會寫測試，為甚麼會沒有測試到呢? 除錯過程也順帶挖出了真相：E2E 早就撞過這個 bug，當時用 `toLocalIso()`（寫入本地 wall time）繞過去了。測試綠了，等於**把錯誤語意寫進測試裡**，幫 bug 蓋了一層厚被子!

## design：AI 列選項，人做決策

病理確立後，讓 AI 列修法。三個選項進了 design.md：

- ❌ **每個讀取點手動補 `Z`**——散彈槍手術，6 個欄位、未來每個新讀取點都要記得，漏一處錯一處
- ❌ **寫入端改存台灣 wall time**——把 workaround 升級成架構，資料從此沒有明確語意
- ✅ **欄位改 `TIMESTAMPTZ`**——修 root cause

第三條路本來以為要搬資料，AI 把 migration 寫出來才發現完全不用：

```sql
ALTER TABLE "AppointmentSlot"
  ALTER COLUMN "startTime" TYPE TIMESTAMPTZ(3) USING "startTime" AT TIME ZONE 'UTC',
  ALTER COLUMN "endTime"   TYPE TIMESTAMPTZ(3) USING "endTime"   AT TIME ZONE 'UTC',
  ALTER COLUMN "createdAt" TYPE TIMESTAMPTZ(3) USING "createdAt" AT TIME ZONE 'UTC';
```

現存的值本來就是 UTC，`USING ... AT TIME ZONE 'UTC'` 只是把語意**標記**上去——零資料搬移。讀回自帶 `+00:00`，`new Date()` 解析正確，**前端顯示程式碼一行都不用改**。

比較三個選項的 diff：選項一要改 N 個讀取點且持續增長，選項二要改寫入端，永久心智負擔，選項三是一段 SQL 加六個 `@db.Timestamptz` 標註。**修 root cause 的 diff，反而最小。**

這裡的分工值得放大來看，AI 三個選項都寫得出來，也都能把任何一個講得頭頭是道。
但是每一次的決策都會影響到這個專案的風格與結構走向，在 AI 把寫程式這個工作做得越來越天衣無縫的時代，我們要做的是提升對架構的判斷力，也是軟體工程師在未來顯現出價值的地方。

## apply：地基驗收完才蓋樓

進到實作，我在 tasks.md 裡把順序卡死：先在 Supabase SQL Editor 跑完 migration、同步 schema.prisma，然後我自己打開畫面，親眼看到原本顯示 02:00 的時段變回 10:00，才准開始動批次功能。這一步沒有自動化，就是人工目視——地基要自己踩過才知道穩不穩。

中間還有一件事差點被漏掉：E2E 裡那個 `toLocalIso()`。root cause 修好之後，它就從「繞過 bug 的權宜之計」變成「每天主動往測試裡寫錯誤語意的地雷」。所以我把「移除它、改回標準 `toISOString()`」也排進 tasks.md，讓 AI 一併處理，49 條 E2E 全量重跑一次綠燈——它必須跟這個 bug 一起消失在同一個 change 裡，留著等「以後有空再清」，通常就是永遠不會清了。

## 覆盤：這個 bug 教了我什麼

照 D19 的覆盤閉環，這次寫進規則庫的有兩條。

第一條：AI 留 workaround 的時候，要逼它寫清楚「為什麼」。這次的 `toLocalIso()` 就是教訓——當初 E2E 撞到時區 bug，AI 為了讓測試先過，自己寫了這個函式繞過去，而我 review 時放行了，沒要求它留任何註解。幾週後 root cause 修好、正確寫法回來了，這個繞路函式反而變成錯的那一個，卻因為沒人記得它只是臨時繞路用的，差點就永遠留在測試裡。所以規則是：AI 可以留 workaround，但我必須要求它註明「為什麼存在、什麼時候該刪」——不寫 code 不代表不把關，這種東西正是人該攔的。

第二條：AI 的推論要逐項驗證過才能放行。前面 AI 把 bug 拆成三個環節——`TIMESTAMP` 欄位、`toISOString()` 寫入、PostgREST 讀回不帶 `Z`——每一個我都查過官方文件、或直接在畫面上驗證過，確認是真的才讓它往下做。這一步不能省：這次三個都對，但假如其中一個是 AI 掰的，我一句「照這個修」下去，動的可是正式資料庫。

## 總結

> AI 把除錯裡「查」的成本降到接近零，於是「判斷」成了人的全部工作——而 SDD 確保每個判斷都留下紀錄、按對的順序執行。
